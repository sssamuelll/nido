# April 2026 Unification Audit

## What this is

A focused refactor sprint across April 2026 that unified five independent sources of repository drift into single, canonical implementations. No new features — pure infrastructure work, all behavior-preserving.

## Why it happened

Nido grew from a weekend prototype into a production household-finance app over the prior six months. That growth left the usual fingerprints: every view re-implemented the same loading/error state machine; route tests carried near-identical helper boilerplate; error handling forked into "use the helper" and "log to console and pray" depending on who wrote the file last; cache state was prop-drilled or simply went stale; and the TypeScript version was pinned to 4.9 even though the tsconfig had moved to 5+ syntax — meaning `tsc` silently failed to run in CI for months.

None of these were urgent bugs. All were the kind of compounding sub-millimeter drift that gets a codebase from "tractable" to "hostile" if left alone.

## The five ejes (axes)

Each merged as its own PR with its own narrative. Listed in the order they shipped.

### Eje E.a — Loading/error UI state ([#223](https://github.com/sssamuelll/nido/pull/223))

Extracted the duplicated `loading` / `error` / `data` triplet from 7+ views into two reusable hooks:

- `useResource<T>(loader, opts?)` — async resource with explicit refetch
- `useAsyncEffect(loader, deps, opts?)` — fire-and-forget loads

Removed the manual `setLoading(true) → try/catch → finally setLoading(false)` ceremony at every call site. Centralized inconsistent error paths (some used `console.error`, some `handleApiError`, one swallowed silently).

### Eje O — Route test helpers ([#224](https://github.com/sssamuelll/nido/pull/224))

Seven `server/routes/*.test.ts` files were carrying near-identical copies of `mockDb`, `getRouteHandler`, and `createResponse`. Extracted into `test/route-helpers.ts`. Each test file keeps a thin local alias to preserve callsite literalness — minimizes diff inside test bodies during the unify.

### Eje I — `handleApiError` silent option ([#225](https://github.com/sssamuelll/nido/pull/225))

The error handler gained an optional `{ silent?: boolean }` parameter that routes errors through the same logging funnel but suppresses the user-facing toast. All Cat 3-auto sites (background loads, polls, mount-time fetches that previously did `catch { console.error(...) }` and orphaned the error) migrated to `handleApiError(err, fallback, { silent: true })`.

Side benefit: the Cat 1-5 error taxonomy is now documented in `AGENTS.md` as a reference for future call sites.

### TypeScript 5.9.3 bump ([#226](https://github.com/sssamuelll/nido/pull/226))

Bumped TS from `^4.9.4` to `^5.9.3`. The repo's `tsconfig.json` already used TS 5+ syntax (`moduleResolution: "bundler"`, `allowImportingTsExtensions`) but `package.json` pinned 4.9, so `tsc` failed at the tsconfig parse step. Without a working `tsc`, the client side never ran a CI typecheck — Vite's `build` uses esbuild and doesn't enforce types.

The bump exposed 44 latent type errors. This PR fixed 14 to keep `npm run build` green:

- Duplicate `handleApiError` import (TS2300)
- Emoji-picker keyword loss due to duplicate object keys (TS1117) — long-standing bug
- `userID` typed as `Uint8Array<ArrayBuffer>` for `@simplewebauthn` (TS 5.7+ tightened the generic)

The remaining 30 are documented as post-audit strictness debt in `docs/HALLAZGOS.md`.

### Eje D — `cacheBus` for cross-view invalidation ([#227](https://github.com/sssamuelll/nido/pull/227))

A tiny pub/sub bus (`src/lib/cacheBus.ts`) for cross-view cache invalidation. `useResource` / `useAsyncEffect` subscribe by canonical key; 27 mutation sites broadcast on success. After this PR, every view subscribed to a canonical key auto-refetches when any other view mutates the underlying resource — without prop-drilling, without lifting state, without context.

Deliberately not TanStack Query. Nido's planned trajectory is local-first (Automerge / P2P sync), and a simple subscribe/invalidate seam is trivial to swap for a `crdtBus` that reacts to document changes; a TanStack Query setup is not. The bus's source file codifies which TQ features are prohibited and the migration trigger ("if 2+ ever needed, migrate").

## Lessons cemented

Three things this audit taught that are now load-bearing for future work:

1. **`npm run build` is the build gate, not `tsc --noEmit -p tsconfig.json`.** The build script runs *both* a server tsc and a vite build, in sequence. Running only the client tsc skipped server-side type errors entirely. We learned this when a CI deploy failed on a server-side `Uint8Array<ArrayBuffer>` regression after we had verified "tsc passes." Now codified in `AGENTS.md` as the local pre-push gate.

2. **Every change to `main` goes through a pull request.** This audit was originally executed as 22 direct pushes to `main` because we'd been doing inter-session "handoffs" via committed docs and bypassing PRs. The result was a commit graph where the last 22 entries lacked PR numbers — visibly different from every preceding commit, and unprofessional to anyone reading the repo from outside. The fix was a hard reset and re-landing all 22 commits as 5 thematic PRs (this audit). The rule is now in `AGENTS.md` with no exceptions: solo, hotfix, docs, one-line — all via PR.

3. **Path-(b) verification per change.** Before/after snapshot of failing-test set + assertion summaries; sorted-set diff must be empty. This pattern (originated in earlier ejes) keeps mechanical refactors honest — if any test transitions FAIL→PASS or PASS→FAIL during a unify, that's a side effect we don't understand and we stop.

## What didn't ship

- **Eje K** (Zod validation on GET query-params) — mechanical, deferred to next sprint.
- **Eje E.b** (error render path consistency: `ErrorView` vs inline `<div>` vs no error UI) — needs UX decisions across 4 views, not mechanical.
- The 30 strictness errors the TS bump exposed that aren't in the build path. Documented in `docs/HALLAZGOS.md` for incremental cleanup.
